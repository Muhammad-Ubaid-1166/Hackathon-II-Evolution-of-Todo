## Overview
This update modifies the FastAPI Todo API to handle todo IDs exclusively on the backend. The frontend will only send the title and category of todos, while the backend manages ID generation and assignment.

## Technology Stack
- Python 3.x
- FastAPI
- SQLModel (for ORM and Pydantic integration)
- SQLAlchemy (underlying ORM)
- PostgreSQL (via psycopg2-binary)
- uv (Python environment and dependency manager)

## Data Model Changes

### TodoCreate
A new input model for creating todos without requiring an ID:

- **title** (string)  
  Title or description of the todo

- **category** (Category enum)  
  Allowed values:
  - backlog
  - todo
  - doing
  - done

The default category value is `backlog`.

### Todo (Existing)
The existing Todo model remains unchanged for database operations and responses:

- **id** (Optional[int])  
  Unique identifier for the todo item, auto-generated by database

- **title** (string)  
  Title or description of the todo

- **category** (Category enum)  
  Allowed values:
  - backlog
  - todo
  - doing
  - done

## API Endpoints Changes

### Create Todo (Updated)
**POST /todos**  
Creates a new todo item in the database using TodoCreate input. Returns the created todo object with auto-generated ID.

### Other Endpoints (Unchanged)
**GET /todos**, **PUT /todos/{id}**, **DELETE /todos/{id}**  
These endpoints remain unchanged, using the existing Todo model for responses and path parameters.

## ID Handling Behavior
- IDs are auto-generated by the database when creating new todos
- Frontend only needs to provide title and category for creation
- Backend ensures ID consistency across all operations
- PUT and DELETE operations use IDs from URL path, not from request body

## Improvements
- Clear separation of concerns between frontend and backend ID management
- Simplified frontend integration by removing ID handling requirements
- Consistent ID generation using database auto-increment
- Better API design following REST principles
